You are a **Senior Full-Stack Software Engineer** improving an existing **Inventory Tracker Application** that was previously generated using a Product Requirements Document (PRD).

Your task is to **extend and fix the current application without breaking existing functionality**.

IMPORTANT GUIDELINES

Before implementing any changes:

1. Review the original PRD.
2. Review the current project structure and codebase.
3. Maintain the current UI design system and architecture.
4. Ensure backward compatibility with existing features.
5. Do NOT rebuild the application — only modify and extend it.

---

SYSTEM CONTEXT

The Inventory Tracker currently includes:

• Dashboard overview
• Inventory list page
• Add inventory item
• Update stock page
• Inventory management tools

However, additional operational features and usability improvements are required.

Implement the following updates.

---

FEATURE 1 — ACTIVITY FOLDER IN SIDEBAR

Add a new section in the **left sidebar navigation** called:

**Report Activity**

The Activity section should function as a **report repository for daily inventory changes**.

When users open this folder, it should display **daily reports showing all inventory modifications made during the day**.

Each report should include:

• Date
• Item name
• SKU
• Quantity change
• Price per unit
• Total value change
• Action type (Add Item / Update Stock / Sale / Edit Item)
• User who made the change (if user tracking exists)

Reports should be grouped by **date**.

Example layout:

Activity
├── 2026-03-15 Report
├── 2026-03-16 Report
├── 2026-03-17 Report

---

FEATURE 2 — AUTOMATIC DAILY REPORT RECORDING

The system must automatically **record all inventory changes** into the Activity reports.

Events that must be logged include:

• Adding a new inventory item
• Updating stock quantity
• Editing item name or category
• Recording sales
• Any inventory adjustments

Each log entry must store:

timestamp
item_id
item_name
action_type
quantity_change
price_per_unit
total_value_change

At the end of each day, logs must be grouped into a **Daily Activity Report**.

---

FEATURE 3 — PASSWORD PROTECTION FOR REPORT ACTIVITY FOLDER

The **Report Activity Folder must be password protected**.

Normal users must still be able to:

• Add inventory
• Update stock
• Record sales
• Edit items

However, **only managers with the correct password can access Report Activity reports**.

Implementation requirements:

When a user clicks the **Report Activity folder**:

1. Display a password prompt modal
2. If password is correct → show Activity reports
3. If password is incorrect → deny access

Security requirements:

• Password must not be stored in plain text
• Use hashed storage if using a backend
• Limit password attempts if possible

Example flow:

Sidebar → Report Activity → Password Prompt → Access Granted → View Reports

---

FEATURE 4 — UPDATE STOCK PAGE SAVE BUTTON

The **Update Stock Page** must include a **Save button**.

Current behavior may automatically update fields, but changes must only be applied when the user clicks **Save** and the page size should be reduced to make the buttons visible for users.

Update Stock Form Fields:

• Quantity
• Price Per Unit
• Total Amount

Calculation Rule:

Total Amount = Quantity × Price Per Unit

Implementation rules:

• Total Amount should auto-calculate in real time
• Total Amount field must be read-only
• Changes must only persist after clicking **Save**

---

FEATURE 5 — ADD NEW INVENTORY ITEM PAGE IMPROVEMENT

Enhance the **Add New Inventory Item Page**.

In addition to the existing fields, include:

• Price Per Unit
• Quantity
• Total Amount

Calculation Rule:

Total Amount = Quantity × Price Per Unit

Implementation rules:

• Total Amount auto-calculates in real time
• Total Amount should be read-only
• Save button must store all values correctly

---

FEATURE 6 — DROPDOWN TEXT VISIBILITY

Improve usability across the entire application.

Requirement:

All dropdown menus must display **text in black font color** to ensure visibility.

Applies to:

• Category dropdown
• Item selection dropdown
• Sales form dropdowns
• Settings dropdowns

Implementation requirements:

• Ensure dropdown text color = black
• Ensure dropdown background provides sufficient contrast
• Ensure readability in both light and dark themes
